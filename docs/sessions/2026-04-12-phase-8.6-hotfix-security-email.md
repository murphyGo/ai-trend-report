## Issues Resolved

### H1 — `search.js` XSS 취약점

**문제**: `innerHTML = results.map(...)`로 카드를 렌더할 때 `escapeHtml`이 `title`과
`summary`에만 적용되고 `url`, `source`, `date`, `report_url`은 raw로 삽입됨.

- Attribute injection: `article.url`에 `"` 포함 시 href 속성 깨짐 → onclick 주입
- `javascript:` URL: 클릭 시 임의 JS 실행
- source/date의 `<script>` 주입

**수정**:
- `safeUrl()` 함수 신설 — `new URL()` 파싱 후 `http:`/`https:`만 허용, 나머지는 `'#'`
- 모든 interpolated 필드에 `escapeHtml` 적용 (url은 `escapeHtml(safeUrl(url))` 이중)
- 상대 경로(`/...`)는 허용

### H2 — `email_notifier.py` HTML f-string escape 부재

**문제**: `_format_article_html`과 `send_error_notification`이 Jinja2 autoescape
없이 f-string으로 HTML을 직접 조립. 악의적 기사 제목이나 에러 메시지에 HTML 태그가
포함되면 그대로 렌더. Gmail은 대부분 `<script>`를 차단하지만 `<img onerror>`,
`<iframe>`, 스타일 주입 등 여러 XSS 벡터 존재.

**수정**:
- `_esc()` 헬퍼 (`html.escape(v, quote=True)`) + None-safe
- `_is_safe_url()` 헬퍼 — http/https 프로토콜만 허용
- title/summary/url/source_name/source_class/error_message 전부 `_esc()` 처리
- `<a href="{url}">` → `<a href="{_esc(url if _is_safe_url(url) else '#')}">`

### H3 — 이메일 수신자 프라이버시 (To 헤더 노출)

**문제**: `msg['To'] = ', '.join(self.recipients)`로 다중 수신자 주소가 서로
노출됨. 프라이버시 유출 + 스팸 학습 위험.

**수정**:
- `msg['To'] = sender` (self-addressed, RFC 준수)
- `msg['Bcc'] = ', '.join(self.recipients)` — 실제 수신자는 Bcc로
- `server.send_message(msg, from_addr=sender, to_addrs=self.recipients)` 명시적
  envelope 전달 (`smtplib`가 Bcc 헤더를 발송 직전 자동 제거)

### H4 — MIMEMultipart('alternative')에 plain 부재

**문제**: `MIMEMultipart('alternative')` 타입인데 HTML 파트만 첨부. RFC 위반 +
일부 스팸 필터 감점 + 텍스트 전용 클라이언트(터미널, 접근성 도구)에서 빈 메시지.

**수정**:
- `_build_plain_message(report)` 신설 — 카테고리별 평문 렌더, quiet-day 배너 포함
- `msg.attach(MIMEText(plain, 'plain', ...))` 를 html 파트 앞에 추가
  (`alternative` 파트 순서는 "덜 선호"부터 "더 선호"로 진행)
- 에러 알림도 평문 대안 포함

### H5 — `sender` 빈 값 fallback

**문제**: `config.email.sender`의 기본값은 `""`이고 GitHub Actions Secrets에
`EMAIL_SENDER`가 없어서 `config.yaml` 없는 환경에서는 빈 From 헤더로 발송 → Gmail
등 대부분 SMTP가 거부.

**수정**:
- `_effective_sender()` 메서드 — `sender or username`
- From 헤더, `from_addr` kwarg 모두 fallback 값 사용
- Gmail처럼 username이 실제 이메일 주소인 경우 자연스러운 fallback

### H6 — `search.js`와 audience 필터 미통합

**문제**: Phase 7에서 audience 필터를 추가했지만 search.js는 `innerHTML`로 카드를
동적 생성. `audience-filter.js`의 `applyFilter`는 DOMContentLoaded 시점에 1회만
실행되므로 이후 동적으로 추가된 카드에는 필터 미적용. 필터 바는 search.html에
포함돼 있는데 실제 효과가 없어 사용자 혼란.

**수정**:

1. `static_generator._generate_search_index` — search-index.json 엔트리에 신규 필드:
   ```json
   {
     "audience": ["DEVELOPER", "ML_EXPERT"],
     "category": "LLM",            // enum name (기존: value였음)
     "category_label": "LLM (대규모 언어 모델)",
     "source": "ARXIV",             // enum name
     "source_label": "arXiv"
   }
   ```

2. `audience-filter.js` — public API 노출:
   ```js
   window.AudienceFilter = {
     apply: function(value) { ... },      // 특정 레벨로 전환
     applyCurrent: function() { ... },    // 저장된 현재 레벨 재적용
     getCurrent: function() { ... },      // 현재 레벨 조회
   };
   ```

3. `search.js` — 렌더 시 `data-audience="..."` 속성 부착 + 렌더 후
   `window.AudienceFilter.applyCurrent()` 호출:
   ```js
   const audienceAttr = Array.isArray(article.audience)
     ? article.audience.join(',')
     : '';
   // ... <article class="article-card" data-audience="${escapeHtml(audienceAttr)}">
   // ... after innerHTML = ...
   if (window.AudienceFilter) window.AudienceFilter.applyCurrent();
   ```

이제 사용자가 필터 바에서 "개발자"를 선택한 뒤 검색을 하면 검색 결과 카드도 해당
필터가 즉시 적용됨.

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| URL safety — 프로토콜 검증 (safeUrl) | `javascript:`/`data:` XSS 벡터 차단. 상대경로(/)는 내부 링크라 허용 |
| HTML escape는 f-string마다 `_esc()` 명시 | Jinja2로 전환은 더 깨끗하지만 범위 큼. 최소 변경으로 동일 효과 |
| Bcc + self-addressed To | 수신자 프라이버시 + 많은 메일 서버가 빈 To를 거부하므로 sender로 채움 |
| plain 파트를 html 앞에 첨부 | RFC 2046 — alternative 파트는 "less preferred"부터 순서대로 |
| sender fallback to username | Gmail 등은 username=이메일주소이므로 합리적 기본값 |
| audience-filter.js에 public API 노출 | IIFE 안에 캡슐화된 함수를 외부에서 호출하려면 명시적 창구 필요. window.AudienceFilter 객체로 최소 3 메서드만 공개 |
| search-index.json의 category/source를 name으로 변경 | audience-filter.js가 enum name 기반이라 일관성. category_label/source_label은 표시용으로 별도 필드 |
| 회귀 방지는 정적 파일 파싱 (jsdom 없이) | 기존 Phase 7.6 패턴과 동일 — JS 런타임 없이 소스 검사 |

## Potential Risks

1. **search-index.json enum name 변경**: 기존 외부 consumer 있으면 breaking. 이
   리포 내부에선 search.js가 유일. 배포 후 첫 빌드가 기존 캐시를 overwrite하므로
   브라우저 캐시 강제 새로고침 필요할 수 있음.
2. **Bcc 방식이 일부 메일 서버에서 의심 점수 증가**: 일부 스팸 필터는 전원 Bcc
   발송을 의심. 특히 대량 발송 시 문제. 본 서비스는 소규모라 허용.
3. **plain text 렌더 품질**: 기본적인 텍스트 포맷이라 UX는 html보다 낮음. 텍스트
   전용 클라이언트 사용자에게는 충분.
